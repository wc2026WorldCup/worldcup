# 世界杯 2026 提醒系统 · 项目说明 & 原理文档

一个 **2026 美加墨世界杯** 的赛程提醒系统,由两部分组成:

1. **一个网页**(实时比分 + 完整赛程 + 直播入口),手机/电脑浏览器打开即用;
2. **一个可订阅的 Apple 日历**(`.ics`),自动把每场比赛的「赛前 15 分钟 / 开球 / 赛果」推送到 iPhone。

> **线上地址**
> - 网页:`https://wc2026worldcup.github.io/worldcup/`
> - 日历订阅:`https://wc2026worldcup.github.io/worldcup/worldcup.ics`

---

## 0. 一句话架构

> **这是一个「纯静态前端」项目:没有后端服务器、没有数据库、没有自建运维。**
> 网页是一个写死了赛程数据的单文件 HTML;实时比分靠浏览器**直接调第三方公开接口**;托管在 **GitHub Pages**(免费静态托管);访问统计用 **Cloudflare Web Analytics**(一段嵌入式 JS)。
> 这套选择让它能**零成本、零维护**地长期运行——这也是面试时最值得讲的核心取舍(见 [§7 技术取舍](#7-关键技术取舍面试重点))。

数据流:

```
                         ┌─────────────────────────┐
   data/matches.json ──► │  构建脚本 (Python, 标准库) │ ──► index.html  ──┐
   (104 场单一数据源)     │  scripts/build_all.py    │ ──► worldcup.ics ─┤
                         └─────────────────────────┘                   │
                                                                       ▼
                                                            GitHub 仓库 (main 分支)
                                                                       │ GitHub Pages 自动发布
                                                                       ▼
   ┌──────────────────── 用户浏览器 ────────────────────┐     ┌──── 用户 iPhone 日历 ────┐
   │ 打开 index.html                                    │     │ 订阅 worldcup.ics        │
   │   • JS 把 104 场数据渲染成卡片                       │     │   • 每 ~12h 回源拉取更新  │
   │   • 按设备时区显示开球时间 (Intl API)               │     │   • 到点弹三道提醒        │
   │   • 直接 fetch ESPN 公开接口拿实时比分 ◄────────────┼─────┼─ 同一份静态文件          │
   │   • Cloudflare 统计 JS 上报浏览量                   │     └─────────────────────────┘
   └────────────────────────────────────────────────────┘
```

## 2. 网页 (`index.html`) 是怎么做的

一个**单文件**网页:HTML 结构 + 内联 CSS + 内联 JavaScript 全在一个文件里。没有打包工具、没有框架(无 React/Vue)、没有外部依赖。打开网页时,浏览器执行内联 JS,把数据渲染成界面。

**数据怎么进网页:** 模板 `src/template.html` 里有一行 `const DATA = __DATA__;`。构建时把 `matches.json` 的全部内容**原样替换**进 `__DATA__`(JSON 本身就是合法的 JS 数组字面量)。所以 **104 场比赛是"写死"在最终 `index.html` 里的**,不是运行时去某个数据库查的。

每场比赛(`matches.json` 里的一条)长这样:

```json
{ "n": 16, "ko": "2026-06-17T01:00:00Z", "st": "小组赛", "sk": "group-stage",
  "g": "J", "h": "Argentina", "a": "Algeria", "hz": "阿根廷", "hf": "🇦🇷",
  "az": "阿尔及利亚", "af": "🇩🇿", "c": "堪萨斯城", "v": "Arrowhead Stadium" }
```

字段:`n`=场次,`ko`=开球时间(UTC),`st`=阶段中文,`sk`=阶段英文 key,`g`=小组,`h/a`=主客队英文,`hz/hf`、`az/af`=中文名+国旗 emoji,`c`=城市中文,`v`=球场。

网页主要逻辑:

- **时区自适应**:开球时间统一存 **UTC**;前端用浏览器原生 `Intl.DateTimeFormat`(`resolvedOptions().timeZone`)换算成**用户设备所在时区**显示。所以同一个网页,中国用户看到北京时间、日本用户看到日本时间,无需后端处理时区。
- **状态机 + 筛选**:每场比赛根据"当前时间 vs 开球时间/是否有比分"判定为 `未开始 / 进行中 / 已结束`;顶部标签可按状态筛选,再叠加 `小组赛 / 淘汰赛` 二级筛选;按"本地日期"分组;首屏自动滚到下一场。
- **实时比分(关键)**:见 [§3](#3-实时比分是怎么来的没有数据库)。
- **直播入口**:见 [§5](#5-直播入口)。
- **设计**:主题「深夜记分牌」,移动端优先;比分用等宽数字;胜方红色「胜」角标、平局蓝色「平」角标;进行中有滚动足球动画。

## 3. 实时比分是怎么来的(没有数据库)

网页**没有自己的比分数据库**。比分是**浏览器直接去调第三方公开接口**拿的,双数据源:

1. **主数据源:ESPN 公开接口**
   `https://site.api.espn.com/apis/site/v2/sports/soccer/fifa.world/scoreboard?dates=YYYYMMDD`
   这个接口 **CORS 开放**,任何网页/任何浏览器都能直接 `fetch`(包括部署到 GitHub Pages 后)。前端对"已开赛的比赛日"按天拉取,把返回的比赛**按队名配对**映射回我们写死的赛程(因为两边英文队名拼写可能不同,代码里维护了一张**别名表**做归一化匹配)。
2. **兜底数据源:Anthropic API + 联网搜索**
   仅在 **Claude 的 Artifact 预览环境内**可用(用于开发时验证),**部署到 GitHub Pages 后这条不生效**,真正生效的是 ESPN。

> **淘汰赛**:在真实对阵确定前,卡片显示占位(如「E组第1 vs A/B/C/D/F组第3」),比分位显示 TBD。详见 [§6](#6-小组赛打完之后淘汰赛对阵怎么更新)。

**为什么用别人的接口而不自己存比分?** 因为自己存比分就需要一个**后端 + 定时任务 + 数据库**去抓取和存储,成本和维护陡增;而这是个一次性赛事项目,直接消费成熟的公开接口,前端零后端就能拿到实时数据。代价见 [§7](#7-关键技术取舍面试重点)。

## 4. 订阅日历 (`worldcup.ics`) 是怎么做的

`.ics` 是 **iCalendar (RFC 5545)** 标准的纯文本日历文件。本项目用 `scripts/build_calendar.py`(纯 Python 标准库,手写文本)从 `matches.json` 生成:

- **208 条日程** = 104 场比赛 × 2:
  - **比赛日程**:两道提醒 `VALARM` —— 开球前 15 分钟(`TRIGGER:-PT15M`)、开球时(`TRIGGER:PT0M`)。
  - **赛果日程**:开球后约 2 小时(小组赛 +135 分钟 / 淘汰赛 +180 分钟)一道提醒,提示去看结果。
- **312 道提醒**(104×2 + 104×1)。
- 时间统一写 **UTC**(`DTSTART:...Z`),Apple 日历自动按设备时区显示。
- 头部声明 `REFRESH-INTERVAL` / `X-PUBLISHED-TTL = PT12H`:提示订阅端每 ~12 小时回源拉取一次更新。
- 每条日程的 `URL` 字段 + 备注里都带链接 `…/worldcup/?from=cal`,既能点回网页,`?from=cal` 又能在统计里标记"从日历来的"访问。

**订阅(Subscribe) vs 导入(Import) —— 一个重要取舍:**

- **订阅**:手机记住的是 URL,会**定期回源**;以后我更新了仓库里的 `.ics`,**所有订阅者自动拿到新版**(适合"淘汰赛对阵会变"的场景)。**代价**:iOS 对**只读订阅日历**里自带的提醒,触发**不稳定**——经常需要手动确保设置正确才弹。
- **导入**:把 `.ics` 一次性导进你自己的日历,提醒**稳定会弹**;**代价**:是快照,以后内容更新了**不会自动同步**,得重新导入。

> **iOS 实测踩坑(写给维护者):** 订阅日历的"提醒"受两处影响——
> ① 设置 → 日历 → 该订阅 → **「日程提醒」要开**(允许这个日历弹提醒);
> ② 同一处 → **订阅详细信息 → 「移除提醒」必须关**(它若开着,会把 `.ics` 里自带的提醒全删掉,于是一条都不弹)。
> 另外 设置 → 日历 → **获取新数据(Fetch New Data)** 别设成"手动",否则订阅不刷新。

## 5. 直播入口

中国大陆 2026 世界杯的**合法转播方**(用于放"看直播"入口):

- **央视(中央广播电视总台)**:大陆地区**独家全媒体转播权**(2026-05-15 官宣)。
- **咪咕(中国移动)**、**小红书**:央视分销的官方持权转播商。
- ⚠️ **抖音这届没有版权**(已退出竞争),所以入口里**没有抖音**。

网页提供两种入口,都指向 **央视频 / 咪咕视频 / 小红书** 三家:

1. **顶部常驻入口**(始终可见),随时可点;
2. **点"进行中"的比赛卡片** → 弹出「选择直播平台」弹窗三选一。

> **限制(诚实说明):** 这三家直播**仅限中国大陆**可看(版权地域限制);链接只能到**平台首页/世界杯专区**,**无法可靠地一键直达"本场直播"**(平台没有可拼接的稳定深链,小红书网页端甚至直达搜索会 404,故指向首页/唤起 App)。

## 6. 小组赛打完之后,淘汰赛对阵怎么更新?

**现状:** 比分已经是**自动**的(ESPN 接口);只有淘汰赛的**对阵队名**需要更新,因为它们写死在静态文件里(`matches.json` → `index.html` 和 `worldcup.ics`)。三档自动化方案,按省事程度递增:

- **档位 0 — 纯手动(最初状态):** 我重新生成 `index.html` + `worldcup.ics`,你上传覆盖**两个文件**。
- **档位 1 — CI 自动构建(本仓库已配好,见 `.github/workflows/build.yml`):** 你只需在 GitHub 上**编辑一个数据文件** `data/matches.json`(把淘汰赛占位换成真实队名)并提交;CI 会**自动重新生成 `index.html` 和 `worldcup.ics` 并提交回仓库**,Pages 随即重新发布,订阅日历下次回源自动更新。**你无需本地构建、无需上传成品。**
- **档位 2 — 全自动抓取(进阶,可选):** 在 CI 里加一步,从 ESPN 公开接口**按开球时间**抓取淘汰赛真实对阵、写回 `matches.json` 再构建,配合定时 `cron` 每日运行。这样**连"填队名"都不用人做**。
  > 注意:这是一次性赛事的一锤子任务,且要等淘汰赛真有数据才能验证脚本是否解析正确;**建议等 6/27 后真有对阵时再实现并实测**,不要盲上线。

**为什么纯静态托管做不到"零干预自动更新"?** 因为 GitHub Pages 只是把仓库里的文件原样发出去,**不会运行你的代码**;要让文件随比赛结果自动变,要么靠**外部定时任务回写仓库**(就是档位 1/2 的 GitHub Actions),要么把队名也改成**前端运行时从接口取**(可行但要重写比分匹配逻辑,且有 ESPN 数据结构变动的风险)。

## 7. 关键技术取舍(面试重点)

> 这一节是面试被追问"为什么这么设计"时的标准答案。每条 = 决策 + 理由 + 代价 + 规模化时怎么改。

**① 纯静态前端 vs 前后端分离**
- 决策:纯静态,无后端无数据库。
- 理由:一次性赛事、个人项目,要**零成本零运维**;GitHub Pages 免费托管静态文件,够用。
- 代价:不能存用户数据、不能做服务端逻辑、实时数据只能靠第三方接口。
- 规模化:若要账户/收藏/推送,会引入轻量后端(如 Serverless 函数 + 托管数据库),前端不变。

**② 实时比分用第三方公开接口(ESPN) vs 自建抓取**
- 决策:前端直接 `fetch` ESPN(CORS 开放)。
- 理由:成熟、免费、零后端即可拿到实时比分。
- 代价:**依赖外部接口的稳定性与字段格式**;接口改版或限流会失效;跨源匹配靠队名(脆弱,故加别名表)。
- 规模化:自己加一层缓存代理(Serverless)统一字段、抗限流、加容错。

**③ 日历"订阅" vs "导入"**
- 决策:默认做成订阅(`REFRESH-INTERVAL`),便于淘汰赛对阵自动下发。
- 代价:iOS 对只读订阅日历的提醒触发不稳定(见 §4 踩坑)。
- 备选:对"提醒必须稳"的用户改用导入(稳定但不自动更新)。这是**可用性 vs 自动性**的权衡。

**④ 赛程数据写死 vs 全量走接口**
- 决策:赛程(队名/时间/场馆)写死在 `matches.json`;只有比分走接口。
- 理由:赛程基本固定,写死可让网页**离线/接口挂了也能看赛程**,首屏快、无加载抖动。
- 代价:淘汰赛对阵变化需重建(见 §6)。

**⑤ 单文件 HTML vs 前端框架/构建链**
- 决策:单文件、原生 JS、无框架、无打包。
- 理由:体量小、零依赖、易托管、易审计、加载快、长期不腐(没有依赖升级负担)。
- 代价:DOM 操作偏手写,逻辑复杂后可维护性不如框架。
- 规模化:界面复杂到一定程度再上框架与组件化。

**⑥ 统计用 Cloudflare Web Analytics(隐私聚合) vs 传统埋点**
- 决策:Cloudflare 一段 JS,无 Cookie、不追踪个人。
- 理由:免费、合规、无需 Cookie 弹窗。
- 代价:**只有聚合的浏览量/访问数,没有"独立访客/设备数"精确值**;且 GitHub Pages 拿不到 `.ics` 被订阅了多少次。
- 备选:想要"独立访客数"可并装 Umami/GoatCounter;`?from=cal` 用来侧面估"从日历点进网页"的量。

**⑦ 用 GitHub 组织(Organization)托管 vs 个人仓库**
- 决策:建一个组织 `wc2026WorldCup` 放仓库。
- 理由:GitHub Pages 的 URL 会暴露用户名;用组织名可**隐藏个人 GitHub 用户名**。
- 备注:`github.io` 会把组织名转小写 → 实际域名是 `wc2026worldcup.github.io`。

## 8. 本地构建 & 部署

**环境:** Python 3(仅标准库,无需 `pip install`)。`.ics` 校验可选装 `icalendar`。

```bash
# 一键构建(生成 index.html 和 worldcup.ics 到项目根目录)
python scripts/build_all.py

# 或分开构建
python scripts/build_site.py        # 只重建网页
python scripts/build_calendar.py    # 只重建日历
```

**部署(GitHub Pages):**
1. 把整个项目推到 GitHub 仓库的 `main` 分支(本项目托管在组织 `wc2026WorldCup` 的 `worldcup` 仓库)。
2. 仓库 Settings → Pages → Source 选 **Deploy from a branch**,分支 `main`,目录 `/(root)`。
3. 等 1–3 分钟,访问 `https://<组织名小写>.github.io/<仓库名>/`。
4. **重新部署时**:覆盖上传(或 `git push`)根目录的 `index.html` / `worldcup.ics` 即可,**URL 不变,对已有用户/订阅者零影响**。

**自动构建(可选,已配好):** 启用仓库的 Actions 后,只要 `push` 改动了 `data/` `src/` `scripts/`,`.github/workflows/build.yml` 会自动重建并提交成品(见 §6 档位 1)。

## 9. 访问统计

- 工具:**Cloudflare Web Analytics**(嵌入式 JS beacon,token 写在 `src/template.html` 的 `</body>` 前)。
- 后台:Cloudflare → Analytics → Web Analytics → 选 `wc2026worldcup.github.io`。
- 能看到:**页面浏览量、访问数、来源、国家/地区、设备类型、路径**。看"大概多少人在看"建议看 **访问量(Visits)** + 调时间范围看趋势。
- 看不到:**独立访客/设备的精确数**(Cloudflare 不追踪个人);**日历被订阅了多少次**(GitHub Pages 无访问日志)。
- `?from=cal`:日历里的链接带这个参数,可在 Cloudflare 的"路径"里找 `/worldcup/?from=cal`,作为"从日历点进来"的侧面信号(若被合并查询参数,可改用独立路径)。

## 10. 已知限制 & 未来可做

- 淘汰赛对阵需更新(见 §6;可上 CI 档位 2 全自动)。
- 直播链接只能到平台,不能直达本场;仅大陆可看。

## 11. 常见面试问答(速查)

- **Q:有后端和数据库吗?** A:没有。纯静态前端 + 第三方公开接口 + 静态托管。这是为"零成本零运维的一次性赛事项目"刻意做的取舍。
- **Q:实时比分哪来的?** A:浏览器直接调 ESPN 的 CORS 开放接口,按比赛日拉取,用别名表把英文队名配对回本地赛程。
- **Q:时区怎么处理的?** A:时间统一存 UTC,前端用 `Intl` API 按设备时区渲染,无需后端。
- **Q:日历为什么有时不弹提醒?** A:iOS 对只读"订阅"日历的提醒触发不稳定,且会被"移除提醒"开关清掉;要稳就改"导入"。这是自动更新 vs 提醒可靠性的权衡。
- **Q:淘汰赛对阵怎么自动更新?** A:已配 GitHub Actions,改一个数据文件即自动重建发布;进一步可让 CI 从接口抓取队名做到零干预。
- **Q:怎么统计有多少人看?** A:Cloudflare Web Analytics 看聚合访问量/趋势;受隐私设计限制,没有精确人数/设备数。
